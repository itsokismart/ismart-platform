## Рабочие шаблоны и чек-листы

### 1. Шаблон для описания Роли (Role)

Используйте эту таблицу для формализации описания ролей в вашей команде или компании. Скопируйте и заполните для каждой ключевой роли.

| Атрибут | Описание |
|---|---|
| `id` | Уникальный идентификатор (например, `backend_engineer_payments`) |
| `name` | Человекочитаемое название («Бэкенд-разработчик в команде платежей») |
| `responsibilities` | Список из 3-5 ключевых зон ответственности |
| `accountable_for_metrics` | Список `id` метрик, за которые отвечает роль |
| `reports_to` | `id` роли руководителя |
| `interacts_with` | Список `id` ролей и контекст взаимодействия |


### 2. Шаблон для описания Процесса (Process)

Используйте эту структуру для детального описания бизнес-процесса. Это основа для его последующей автоматизации.

```yaml
# --- Описание процесса ---
id: # уникальный_идентификатор_процесса
name: # Название процесса
owner: # id_роли_владельца
trigger: # Событие, которое запускает процесс
result: # Конечный результат выполнения процесса

# --- Шаги процесса ---
steps:
  - name: # Название шага 1
    description: # Краткое описание шага
    responsible_role: # id_ответственной_роли
    input: # Входные данные/артефакты
    output: # Выходные данные/артефакты
    tools: # Список инструментов

  - name: # Название шага 2
    description: # ...
    responsible_role: # ...
    input: # ...
    output: # ...
    tools: # ...
```

### 3. Шаблон для описания Метрики (Metric)

Используйте эту таблицу для создания единого реестра бизнес-метрик.

| Атрибут | Описание |
|---|---|
| `id` | Уникальный идентификатор (например, `user_acquisition_cost`) |
| `name` | Название метрики («Стоимость привлечения пользователя») |
| `description` | Что измеряет метрика и почему она важна |
| `formula` | Однозначная формула для расчета |
| `data_source` | Источник(и) данных для расчета |
| `owner` | `id` роли, ответственной за метрику |
| `related_goals` | Список `id` бизнес-целей, на которые влияет метрика |

### 4. Чек-лист для самопроверки готовности онтологии

Используйте этот чек-лист, чтобы убедиться, что ваши описания готовы для использования AI-агентами.

**Сущности:**
- [ ] Определены ли все ключевые роли в рамках выбранной области?
- [ ] Описаны ли для каждой роли ее зоны ответственности и связи?
- [ ] Определены ли основные бизнес-процессы?
- [ ] Описаны ли для каждого процесса его владелец, триггер и результат?
- [ ] Определены ли ключевые метрики и показатели эффективности?
- [ ] Описаны ли для каждой метрики формула расчета и источник данных?

**Связи:**
- [ ] Установлена ли связь «руководитель-подчиненный» (`reports_to`) для всех ролей?
- [ ] Определены ли владельцы (`owner`) для всех процессов и метрик?
- [ ] Связаны ли метрики с бизнес-целями (`related_goals`)?
- [ ] Прослеживается ли логическая цепочка от шагов процесса к его конечному результату?
- [ ] Можно ли по графу понять, как изменение одной сущности повлияет на другие?

**Качество описания:**
- [ ] Используются ли уникальные идентификаторы (`id`) для всех сущностей?
- [ ] Являются ли описания однозначными и понятными для нового сотрудника?
- [ ] Выбран ли единый формат для описаний (например, YAML или JSON)?
- [ ] Готовы ли вы поддерживать эту онтологию в актуальном состоянии?

### 5. Фреймворк: Матрица оценки зрелости онтологии

Этот фреймворк поможет оценить, на каком уровне зрелости находится ваша компания в вопросе использования онтологий.

| Уровень | Описание | Характеристики | Действия для перехода на следующий уровень |
|---|---|---|---|
| **0: Хаос** | Данные полностью децентрализованы и неструктурированы. Знания хранятся «в головах» сотрудников. | - Документы хранятся на локальных дисках.<br>- Нет единых справочников.<br>- Поиск информации занимает часы. | - Создать централизованное хранилище (Wiki, Confluence).<br>- Начать документировать базовые процессы. |
| **1: Структуризация** | Появляются первые структурированные источники данных (CRM, таск-трекеры). Знания начинают формализоваться. | - Есть CRM, Jira, HR-система.<br>- Данные внутри каждой системы структурированы.<br>- Связи между системами отсутствуют. | - Начать создавать единые справочники (например, список ролей).<br>- Описать 2-3 ключевых сквозных процесса. |
| **2: Описание связей** | Компания начинает осознанно описывать связи между сущностями из разных систем. Появляются первые элементы онтологии. | - Создаются машиночитаемые описания ролей, процессов.<br>- Появляются первые дашборды, объединяющие данные из разных источников.<br>- Внедряются уникальные ID для сущностей. | - Построить первую версию контекстного графа.<br>- Начать использовать AI-агентов для простых запросов к этому графу. |
| **3: Семантическая сеть** | Онтология становится ядром управления знаниями. AI-агенты способны на сложные логические выводы. | - Контекстный граф компании покрывает большинство доменов.<br>- Онтология активно используется для принятия решений.<br>- AI-агенты проактивно отслеживают изменения и инициируют процессы. | - Интегрировать онтологию со всеми новыми системами.<br>- Развивать предиктивные возможности AI-агентов на основе графа. |
