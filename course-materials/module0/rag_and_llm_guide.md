# RAG и работа с большими языковыми моделями (LLM)

Этот документ представляет собой углубленное руководство по технологии RAG (Retrieval-Augmented Generation) и практическим аспектам работы с большими языковыми моделями. Мы рассмотрим ключевые концепции, сравним подходы к оптимизации, изучим реальные кейсы внедрения и разберем пошаговый процесс создания RAG-пайплайна.

## 1. RAG или Файн-тюнинг LLM: Как выбрать?

Когда базовая языковая модель (LLM) не справляется со специфическими задачами или не знает актуальных фактов, перед разработчиками встает выбор между двумя основными подходами: RAG и файн-тюнингом (Fine-Tuning) [1].

**RAG (Retrieval-Augmented Generation)** соединяет LLM с внешними источниками знаний (базами данных, документами) без изменения внутренней структуры и весов самой модели. При каждом запросе система сначала ищет релевантную информацию в базе, а затем передает ее модели вместе с запросом пользователя.

**Файн-тюнинг (Fine-Tuning)** — это процесс дообучения модели на узком, специализированном наборе данных. В этом случае изменяются внутренние параметры (веса) нейросети, чтобы она "впитала" новые знания или стиль общения.

| Характеристика | RAG (Генерация с поиском) | Fine-Tuning (Файн-тюнинг) |
| :--- | :--- | :--- |
| **Принцип работы** | Динамический поиск внешних данных | Изменение внутренних весов модели |
| **Актуальность данных** | Высокая. Базу знаний можно обновить за секунды без переобучения модели | Низкая. Требует повторного дорогостоящего дообучения при изменении фактов |
| **Стоимость внедрения** | Относительно низкая (требует настройки БД и пайплайнов) | Высокая (требует мощных GPU и качественных датасетов) |
| **Борьба с галлюцинациями** | Эффективна (модель опирается на найденные достоверные факты) | Менее эффективна (модель может путать заученные факты) |
| **Идеальный сценарий** | Корпоративный поиск, ответы по документам, работа с динамичными данными | Обучение специфическому стилю, формату (например, писать код на редком языке) |

**Вывод:** RAG является предпочтительным выбором для большинства корпоративных задач, где важна точность фактов и возможность быстро обновлять информацию. Файн-тюнинг лучше использовать, когда нужно изменить само *поведение* или *стиль* модели [2].

## 2. Ключевые концепции RAG

Для понимания того, как работает RAG, необходимо разобраться в трех фундаментальных концепциях.

### 2.1. Эмбеддинги (Embeddings)

Эмбеддинги — это числовые векторы, представляющие текст в многомерном пространстве. Они преобразуют слова, предложения или целые документы в формат, понятный компьютеру. Главная особенность эмбеддингов заключается в том, что тексты с похожим смыслом будут находиться близко друг к другу в этом векторном пространстве [3]. Например, векторы фраз "собака бежит" и "пес мчится" будут почти идентичны, хотя слова в них разные.

### 2.2. Векторная база данных

Векторная база данных (например, FAISS, ChromaDB, Pinecone, Qdrant) — это специализированное хранилище для эмбеддингов. В отличие от традиционных реляционных БД (SQL), которые ищут точные совпадения, векторные БД оптимизированы для быстрого поиска "ближайших соседей" (Nearest Neighbor Search). Они используют математические метрики, такие как косинусное расстояние (cosine similarity), чтобы мгновенно находить фрагменты текста, наиболее релевантные запросу пользователя [4].

### 2.3. Семантический поиск

Семантический поиск — это технология поиска по *смыслу* и контексту, а не по точному совпадению ключевых слов.

*   **Полнотекстовый поиск (Традиционный):** Сканирует все слова в документах, использует инвертированные индексы (как в Elasticsearch). Он эффективен для огромных коллекций, но "слеп" к контексту. Если вы ищете "проблемы с Apple", он может выдать статьи о гнилых яблоках, а не о смартфонах.
*   **Семантический поиск (Векторный):** Понимает намерение пользователя благодаря NLP и эмбеддингам. Он обрабатывает сложные запросы на естественном языке и находит правильный ответ, даже если в тексте нет ни одного слова из исходного запроса.

## 3. Реальные примеры использования RAG в бизнесе

Многие ведущие компании уже успешно внедрили RAG для оптимизации своих процессов [5]:

*   **DoorDash:** Создали чат-бота для поддержки курьеров (Dashers). Система сжимает историю диалога, ищет решения в корпоративной базе знаний и генерирует ответ. Внедрены "LLM Guardrails" для предотвращения галлюцинаций и нарушений политик компании.
*   **LinkedIn:** Внедрили RAG в сочетании с графами знаний (Knowledge Graphs) для службы поддержки. Это сократило среднее время решения проблемы на 28.6%.
*   **Thomson Reuters:** Улучшили обслуживание клиентов. Эмбеддинги используются для поиска по внутренней базе документов, после чего LLM формирует структурированный ответ, снижая риск ошибок.
*   **Pinterest:** Используют RAG для помощи аналитикам в написании SQL-запросов (Text-to-SQL). Система ищет релевантные таблицы в векторном индексе и передает их LLM для формирования правильного запроса.
*   **Яндекс (Neurosupport):** Сервис генерирует умные подсказки для операторов контакт-центра. Благодаря RAG система эффективно решает узкопрофильные задачи, снизив время обработки обращений в Яндекс Маркете на 15%.
*   **Grab:** Азиатский супер-апп автоматизировал создание аналитических отчетов о мошенничестве. RAG-агент ищет нужные SQL-скрипты, выполняет их и суммаризирует результаты, экономя аналитикам 3-4 часа на каждый отчет.

## 4. Внедрение RAG в частной компании

Внедрение RAG-систем приносит ощутимую пользу корпоративному сектору за счет:
*   Интеграции с разрозненными корпоративными базами знаний (регламенты, PDF-отчеты, переписка в Slack/Teams, конфлюенс).
*   Ускорения работы техподдержки, автоматической проверки страховых полисов и анализа сложных юридических документов.
*   Колоссальной экономии времени высокооплачиваемых экспертов (юристов, инженеров, аналитиков).
*   Автоматизации HR-процессов (быстрый онбординг новичков, ответы на вопросы по отпускам и льготам).

## 5. Пошаговое руководство: Создание RAG-пайплайна

Ниже представлен пошаговый пример создания локального RAG-пайплайна на Python с использованием LLM (например, через Ollama) и векторной базы FAISS [6].

### Шаг 1. Загрузка документов

Сначала необходимо извлечь текст из исходных файлов (.md, .pdf, .txt).
*   Для PDF используется библиотека `pdfminer.high_level` (`extract_text_from_pdf`) или `PyPDF2`.
*   Для Markdown — библиотека `markdown`.
*   Текст необходимо очистить от лишних пробелов, табуляций и HTML-тегов.

### Шаг 2. Разбивка на фрагменты (Chunking)

LLM имеют ограничение на размер контекстного окна, поэтому большие документы нужно разбить на смысловые фрагменты (chunks).
*   **Оптимальный размер фрагмента:** около 1000–1200 символов (или токенов).
*   **Перекрытие (Overlap):** 200–300 символов. Конец одного фрагмента должен частично повторяться в начале следующего, чтобы не потерять контекст на стыке предложений.
*   Каждому фрагменту добавляются метаданные (источник, номер страницы) для улучшения фильтрации.

### Шаг 3. Векторизация (Эмбеддинги)

Текстовые фрагменты преобразуются в многомерные векторы с помощью моделей встраивания (например, `nomic-embed-text` через Ollama или модели от Hugging Face). Полученные векторы индексируются и сохраняются в векторной базе данных (например, FAISS или ChromaDB) для обеспечения молниеносного поиска.

### Шаг 4. Настройка LLM

Выбирается и загружается языковая модель.
*   Для локального использования отлично подходят Llama 3, Mistral или Gemma через платформу Ollama.
*   Настраиваются параметры генерации: `temperature` (рекомендуется низкая, около 0.1, для строгих фактологических ответов), максимальная длина ответа.

### Шаг 5. Обработка запросов и поиск

Когда пользователь задает вопрос (например: "Найди условия оплаты в договоре"):
1.  Сам запрос преобразуется в вектор с помощью той же модели эмбеддингов.
2.  Векторная БД (FAISS) вычисляет косинусное расстояние и возвращает топ-K (например, топ-3 или топ-5) наиболее релевантных фрагментов из базы документов.

### Шаг 6. Генерация ответа (Augmentation & Generation)

Найденные фрагменты объединяются с исходным вопросом пользователя в единый промпт (системное сообщение).
*Пример системного промпта:* "Используя только следующий контекст, ответь на вопрос пользователя. Если ответа нет в контексте, скажи 'Я не знаю'. Контекст: [вставка найденных фрагментов]".
LLM получает этот обогащенный промпт и генерирует точный ответ на основе предоставленных фактов.

### Шаг 7. Постобработка и интеграция

*   **Валидация:** Ответ очищается от лишних рассуждений модели. Проверяется корректность форматов (ИНН, даты, суммы).
*   **Интеграция:** Данные экспортируются в JSON, создается API для вызова системы из ERP, 1С, DWH или корпоративного мессенджера.
*   **Безопасность:** При работе с конфиденциальными документами (NDA, финансовые отчеты) весь пайплайн (включая LLM и векторную БД) разворачивается на локальных серверах компании без доступа в интернет.

**Итог:** Правильно настроенный RAG-пайплайн способен автоматизировать извлечение данных из документов с точностью до 99.7%, кардинально снижая нагрузку на специалистов и ускоряя бизнес-процессы.
